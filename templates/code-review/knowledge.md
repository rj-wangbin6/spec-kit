# Code Review 问题分类标准

## 文档目标

**本文档提供问题分类的判断依据，确保AI对相同类型的问题总能归类到同一个分类。**

**核心原则**：
- 基于问题的**本质影响**而非表面现象分类
- 优先级：安全 > 功能 > 性能 > 质量 > 风格
- 当问题同时符合多个分类时，选择优先级最高的

**版本**：v2.0  
**更新时间**：2026-03-06

---

## 分类决策流程

```
问题识别
    ↓
是否涉及安全风险？ → 是 → 🔒 安全问题
    ↓ 否
是否影响功能正确性？ → 是 → 📌 功能问题
    ↓ 否
是否影响系统性能/资源？ → 是 → ⚡ 性能问题
    ↓ 否
是否影响代码可维护性？ → 是 → 📝 代码质量
    ↓ 否
是否仅影响代码格式/风格？ → 是 → 🎨 代码风格
```

---

## 分类1：📌 功能问题

### 核心判断标准

**关键问题：这个问题会导致功能不正确吗？**

如果答案是"是"，则归类为功能问题。

### 判断依据

| 判断维度 | 是否属于功能问题 |
|---------|----------------|
| **影响功能正确性** | ✅ 是 |
| **导致业务逻辑错误** | ✅ 是 |
| **在特定场景下功能失败** | ✅ 是 |
| **不影响功能，仅影响性能** | ❌ 否（性能问题） |
| **不影响功能，仅影响安全** | ❌ 否（安全问题） |
| **不影响功能，仅代码质量差** | ❌ 否（质量问题） |

### 子类型判断

#### 1. 逻辑错误

**核心特征**：
- ✅ 计算结果错误
- ✅ 条件判断错误
- ✅ 状态流转错误
- ✅ 业务规则实现错误

**关键词**：
- 计算错误、公式错误、算法错误
- 条件写反、判断错误
- 状态不符、流程错误

**典型场景**：
```
❌ 金额计算：price * discount（错误，折扣应该是 price * (1 - discount)）
❌ 条件判断：age >= 18（错误，需求是 > 18）
❌ 状态判断：允许已取消订单继续支付
```

**边界说明**：
- 如果是安全相关的逻辑错误 → 归为**安全问题**
- 如果是性能相关的算法选择 → 归为**性能问题**

---

#### 2. 边界条件

**核心特征**：
- ✅ 未处理 null/空值
- ✅ 未处理空集合/空字符串
- ✅ 未处理边界值（0、负数、最大/最小值）
- ✅ 可能数组越界
- ✅ 可能除零错误

**关键词**：
- NullPointerException、空指针
- 数组越界、IndexOutOfBounds
- 除零、DivideByZero
- 空字符串、空集合

**典型场景**：
```
❌ user.getName()  // user可能为null
❌ list.get(0)  // list可能为空
❌ amount / count  // count可能为0
❌ array[index]  // index可能越界
```

**边界说明**：
- 即使是因为缺少参数校验导致的 → 仍归为**边界条件**
- 如果边界问题导致安全风险 → 归为**安全问题**

---

#### 3. 异常处理

**核心特征**：
- ✅ 未捕获应该捕获的异常
- ✅ 捕获异常但不处理（空catch块）
- ✅ 异常信息不明确
- ✅ 事务未正确回滚
- ✅ 资源未在finally释放

**关键词**：
- 空catch块、吞异常
- 异常未捕获
- 事务未回滚
- 资源泄漏（如果是性能影响 → 性能问题）

**典型场景**：
```
❌ 调用可能抛异常的方法但未try-catch
❌ catch (Exception e) { }  // 空块
❌ catch后未回滚事务
❌ 异常信息不明确：throw new Exception("error")
```

**边界说明**：
- 如果是资源未释放导致内存泄漏 → 归为**性能问题**
- 如果是异常暴露敏感信息 → 归为**安全问题**

---

#### 4. 功能遗漏

**核心特征**：
- ✅ 需求中的功能未实现
- ✅ 缺少必要的参数校验
- ✅ 遗漏某个业务分支
- ✅ 缺少必要的日志（影响问题排查）
- ✅ 未实现幂等性（应该实现但未实现）

**关键词**：
- 功能缺失、未实现
- 参数未校验
- 分支遗漏
- 幂等性缺失

**典型场景**：
```
❌ 接口缺少参数校验
❌ 支付接口未实现幂等性
❌ 遗漏了"已退款"状态的处理分支
❌ 核心业务缺少操作日志
```

**边界说明**：
- 如果是非必要的功能优化建议 → 归为**代码质量**
- 如果是缺少权限校验 → 归为**安全问题**

## 分类2：🔒 安全问题

### 核心判断标准

**关键问题：这个问题会造成安全风险吗？**

如果答案是"是"，则**优先**归类为安全问题（即使同时也是功能问题）。

### 判断依据

| 判断维度 | 是否属于安全问题 |
|---------|----------------|
| **可被恶意利用** | ✅ 是 |
| **导致数据泄露** | ✅ 是 |
| **导致权限绕过** | ✅ 是 |
| **不当的加密/存储** | ✅ 是 |
| **仅是功能错误，无安全风险** | ❌ 否（功能问题） |
| **仅影响性能** | ❌ 否（性能问题） |

### 优先级规则

**安全问题优先级最高**，如果一个问题同时符合"功能问题"和"安全问题"，归为**安全问题**。

例如：缺少权限校验
- 可以说是"功能遗漏"（缺少校验功能）
- 但因为会导致越权访问 → 归为**安全问题-权限校验**

### 子类型判断

#### 1. 注入攻击

**核心特征**：
- ✅ SQL字符串拼接
- ✅ HTML未转义输出到页面
- ✅ 命令行参数未转义
- ✅ XML/JSON注入
- ✅ 模板注入

**关键词**：
- SQL拼接、字符串拼接SQL
- XSS、跨站脚本
- 命令注入、代码注入
- 参数未转义、未过滤

**必须满足**：用户输入可以影响执行逻辑

**典型场景**：
```
✅ 安全问题：
  "SELECT * FROM users WHERE name = '" + userName + "'"
  // userName来自用户输入，可注入

✅ 安全问题：
  response.write("<div>" + userInput + "</div>")
  // userInput未转义，可XSS

❌ 不是注入（仅是代码质量差）：
  "SELECT * FROM users WHERE name = '" + CONSTANT_NAME + "'"
  // CONSTANT_NAME是常量，不可控
```

**边界说明**：
- 如果拼接的是常量/配置 → **不是安全问题**，归为**代码质量-重复代码**
- 必须是**用户可控的输入**才是注入风险

---

#### 2. 权限校验

**核心特征**：
- ✅ 接口/方法缺少权限检查
- ✅ 可以访问/修改他人数据
- ✅ Token/Session未校验
- ✅ 身份认证缺失
- ✅ 权限校验可被绕过

**关键词**：
- 越权、权限绕过
- 未校验token、未校验登录
- 水平越权、垂直越权
- 缺少权限检查

**必须满足**：会导致未授权访问

**典型场景**：
```
✅ 安全问题：
  @GetMapping("/order/{id}")
  public Order get(@PathVariable Long id) {
      return orderDao.findById(id);
  }
  // 未检查订单是否属于当前用户 → 越权

✅ 安全问题：
  public void delete(Long id) {
      if (!isAdmin()) return;  // 仅前端校验
      dao.delete(id);
  }
  // 权限校验可被绕过

❌ 不是权限问题（是功能遗漏）：
  缺少参数校验，但不涉及权限
```

**边界说明**：
- 如果是缺少**参数校验**（非权限） → **功能问题-功能遗漏**
- 如果是缺少**权限校验** → **安全问题-权限校验**

---

#### 3. 敏感信息

**核心特征**：
- ✅ 密码明文存储
- ✅ 密码明文传输
- ✅ 日志打印敏感信息
- ✅ 响应中返回敏感信息
- ✅ 密钥/Token硬编码
- ✅ 敏感信息未脱敏

**关键词**：
- 密码明 文、明文密码
- 日志泄露、信息泄露
- 硬编码密钥、硬编码token
- 敏感信息、个人隐私

**敏感信息定义**：
- 密码、密钥、Token
- 身份证、手机号、银行卡号
- 地址、姓名（根据隐私政策）

**典型场景**：
```
✅ 安全问题：
  log.info("用户登录：{}, 密码：{}", user, password)
  // 日志包含密码

✅ 安全问题：
  user.setPassword(rawPassword);
  userDao.save(user);
  // 密码明文存储

✅ 安全问题：
  public static final String API_KEY = "abc123...";
  // 密钥硬编码

❌ 不是敏感信息（是代码质量）：
  log.info("处理订单：{}", orderId)
  // orderId不敏感
```

**边界说明**：
- 判断依据：**该信息泄露会造成安全风险吗？**
- 如果会 → 安全问题
- 如果只是代码不规范 → 代码质量

---

#### 4. 加密问题

**核心特征**：
- ✅ 使用弱加密算法（MD5、SHA1用于密码）
- ✅ 敏感数据明文传输
- ✅ 加密密钥管理不当
- ✅ 随机数生成不安全（Random代替SecureRandom）
- ✅ 加密实现错误

**关键词**：
- MD5加密密码、SHA1加密密码
- 明文传输
- 弱加密算法
- Random、Math.random用于安全场景

**典型场景**：
```
✅ 安全问题：
  String hash = MD5.encrypt(password);
  // MD5用于密码 → 弱加密

✅ 安全问题：
  String token = String.valueOf(new Random().nextInt());
  // Random不安全，应用SecureRandom

❌ 不是安全问题（取决于场景）：
  String hash = MD5.hash(fileContent);
  // MD5用于文件完整性校验 → 可接受
```

**边界说明**：
- **关键看用途**：
  - MD5用于密码 → 安全问题
  - MD5用于文件校验 → 不是安全问题
  - Random用于生成Token → 安全问题
  - Random用于抽奖 → 不是安全问题

---

## 分类3：⚡ 性能问题

### 核心判断标准

**关键问题：这个问题会导致性能下降或资源浪费吗？**

如果答案是"是"，且不涉及功能/安全问题，则归类为性能问题。

### 判断依据

| 判断维度 | 是否属于性能问题 |
|---------|----------------|
| **导致响应变慢** | ✅ 是 |
| **导致内存/CPU占用过高** | ✅ 是 |
| **导致资源泄漏** | ✅ 是 |
| **数据库查询效率低** | ✅ 是 |
| **功能正确但效率低** | ✅ 是 |
| **功能不正确** | ❌ 否（功能问题） |
| **仅是代码不优雅** | ❌ 否（代码质量） |

### 子类型判断

#### 1. 查询优化

**核心特征**：
- ✅ N+1查询（循环中查数据库）
- ✅ 缺少索引导致全表扫描
- ✅ 一次查询返回过多数据
- ✅ 未使用批量操作
- ✅ 不必要的关联查询

**关键词**：
- N+1查询、循环查询
- 全表扫描、缺少索引
- 慢查询、查询优化
- 批量操作

**必须满足**：涉及数据库/缓存/外部接口查询

**典型场景**：
```
✅ 性能问题：
  for (Order order : orders) {
      User user = userDao.findById(order.getUserId());
  }
  // N+1查询

✅ 性能问题：
  SELECT * FROM orders WHERE status = 'xxx'
  // status字段无索引，100万数据全表扫描

✅ 性能问题：
  SELECT * FROM orders
  // 返回全部字段和全部数据，实际只需要部分

❌ 不是查询问题（是代码质量）：
  重复的查询逻辑代码，但执行次数不多
```

**判断标准**：
- **数量级**：循环次数/数据量是否可能很大？
  - 循环10次查询 → 可能不是大问题
  - 循环未知次数（来自用户数据） → 性能问题
- **是否必要**：能否通过批量/JOIN优化？

---

#### 2. 内存泄漏

**核心特征**：
- ✅ 资源未关闭（流、连接、文件）
- ✅ 静态集合持续增长
- ✅ 大对象长期缓存且不清理
- ✅ 线程/线程池未正确管理
- ✅ 监听器未移除

**关键词**：
- 资源泄漏、内存泄漏
- 未关闭、未释放
- 静态集合、全局缓存
- 线程泄漏

**必须满足**：会导致资源持续占用

**典型场景**：
```
✅ 性能问题：
  FileInputStream fis = new FileInputStream(file);
  // 使用但未关闭

✅ 性能问题：
  private static Map<String, Object> cache = new HashMap<>();
  public void add(String key, Object value) {
      cache.put(key, value);  // 只放不清，持续增长
  }

✅ 性能问题：
  while (true) {
      new Thread(() -> {}).start();  // 线程不断创建
  }

❌ 不是内存泄漏（是功能问题）：
  未关闭资源导致功能失败（如文件锁）
```

**边界说明**：
- 如果资源未关闭**导致功能错误** → **功能问题-异常处理**
- 如果资源未关闭**仅影响性能/资源** → **性能问题-内存泄漏**

---

#### 3. 无效循环

**核心特征**：
- ✅ 循环中重复计算不变值
- ✅ 不必要的嵌套循环
- ✅ 可以用流式API优化
- ✅ 循环中创建大量临时对象

**关键词**：
- 循环优化、重复计算
- 嵌套循环、暴力循环
- 效率低、时间复杂度高

**典型场景**：
```
✅ 性能问题：
  for (int i = 0; i < list.size(); i++) {
      // 每次循环都调用size()
  }

✅ 性能问题：
  for (int i = 0; i < n; i++) {
      String result = new String(CONSTANT);  // 每次创建
  }

✅ 性能问题：
  for (User u1 : users) {
      for (User u2 : users) {  // O(n²)，可能可以优化
      }
  }

❌ 不是性能问题（是代码质量）：
  循环次数很少（< 100），性能影响可忽略
```

**判断标准**：
- **是否可优化？** 有更高效的算法
- **数量级大吗？** 数据量可能很大
- **影响明显吗？** 会造成明显延迟

---

#### 4. 缓存问题

**核心特征**：
- ✅ 应该缓存但未缓存（频繁查询不变数据）
- ✅ 缓存穿透风险（null值未缓存）
- ✅ 缓存雪崩风险（同时过期）
- ✅ 缓存更新策略不当
- ✅ 缓存击穿（热点数据未加锁）

**关键词**：
- 未缓存、应缓存
- 缓存穿透、缓存雪崩、缓存击穿
- 缓存更新、缓存失效

**典型场景**：
```
✅ 性能问题：
  public Dict getDict(String code) {
      return dictDao.findByCode(code);
  }
  // 字典数据不变，但每次都查库

✅ 性能问题：
  User user = cache.get(userId);
  if (user == null) {
      user = db.query(userId);  // 未判断是否真的不存在
      cache.put(userId, user);
  }
  // 缓存穿透：查不存在的userId会一直打到DB

❌ 不是缓存问题（是代码质量）：
  数据不适合缓存（实时性要求高）
```

**判断标准**：
- **是否频繁访问？** 查询QPS高
- **是否适合缓存？** 数据变化不频繁
- **有无风险？** 缓存三大问题

---

## 分类4：📝 代码质量

### 核心判断标准

**关键问题：这个问题影响代码可维护性吗？**

如果答案是"是"，且不影响功能/安全/性能，则归类为代码质量。

### 判断依据

| 判断维度 | 是否属于代码质量 |
|---------|----------------|
| **影响可维护性** | ✅ 是 |
| **影响可扩展性** | ✅ 是 |
| **增加理解难度** | ✅ 是 |
| **违反设计原则** | ✅ 是 |
| **影响功能正确性** | ❌ 否（功能问题） |
| **仅是格式/风格** | ❌ 否（代码风格） |

### 子类型判断

#### 1. 重复代码

**核心特征**：
- ✅ 相同或相似代码多处出现
- ✅ 可以提取为方法但未提取
- ✅ 复制粘贴的代码块
- ✅ 违反DRY原则

**关键词**：
- 重复代码、代码重复
- DRY、复制粘贴
- 可提取方法

**判断标准**：
- **连续3行以上**相同/高度相似代码
- 出现在**2处以上**
- **可以提取**为公共方法/工具类

**典型场景**：
```
✅ 代码质量问题：
  // 方法A
  if (status == 1 || status == 2) {
      throw new Exception("不可操作");
  }
  
  // 方法B
  if (status == 1 || status == 2) {
      throw new Exception("不可操作");
  }
  // 相同逻辑重复

✅ 代码质量问题：
  // 3个方法中都有相同的参数校验代码

❌ 不是重复代码：
  int sum1 = a + b;  // 场景1
  int sum2 = c + d;  // 场景2
  // 形式相同，但语义不同，不应该合并
```

**边界说明**：
- 判断依据：**语义是否相同**
- 如果只是形式像但含义不同 → 不是重复代码
- 如果语义相同但场景不同 → 仍是重复代码，应提取

---

#### 2. 复杂度高

**核心特征**：
- ✅ 圈复杂度过高（> 10）
- ✅ 方法过长（> 50行）
- ✅ 嵌套过深（> 3层）
- ✅ 参数过多（> 5个）
- ✅ 类职责过多

**关键词**：
- 圈复杂度、复杂度
- 方法过长、函数过长
- 嵌套过深、if嵌套
- 参数过多

**判断标准**：
- 圈复杂度：超过10
- 方法行数：超过50行
- 嵌套层级：if/for超过3层
- 参数数量：超过5个
- 类行数：超过500行

**典型场景**：
```
✅ 代码质量问题：
  public void process(...) {  // 100行
      if (a) {
          if (b) {
              if (c) {
                  if (d) {  // 嵌套4层
                  }
              }
          }
      }
  }

✅ 代码质量问题：
  public void create(String a, String b, int c, int d, 
                     boolean e, double f, long g) {
      // 7个参数
  }

❌ 不是复杂度问题：
  // 30行方法，逻辑清晰，职责单一
```

**边界说明**：
- **数值仅供参考**，具体根据实际情况判断
- 关键看：**是否难以理解、难以测试**

---

#### 3. 命名问题

**核心特征**：
- ✅ 无意义的命名（a, b, data, temp, obj）
- ✅ 命名与实际功能不符
- ✅ 缩写过度或不一致
- ✅ 过于宽泛的命名

**关键词**：
- 命名不规范、命名不明确
- 无意义命名、语义不清
- 缩写不当

**典型场景**：
```
✅ 代码质量问题：
  List<Object> data = getData();
  // data是什么数据？

✅ 代码质量问题：
  void process(int a, int b) {
  // a和b是什么？

✅ 代码质量问题：
  boolean flag = checkStatus();
  // flag表示什么？

✅ 代码质量问题：
  User usr;  // 缩写
  User user;  // 全称
  // 同一个类中缩写不一致

❌ 不是命名问题（是风格问题）：
  int userAge;  // 应该是 age（在User类中）
  // 这是命名风格选择，不影响理解
```

**判断标准**：
- **是否影响理解？**
  - `data` → 影响理解，什么数据？
  - `orderList` → 不影响理解
- **是否语义不符？**
  - `getUser()` 实际是创建用户 → 语义不符
  - `createUser()` 创建用户 → 语义相符

#### 4. 设计原则

**核心特征**：
- ✅ 单一职责原则（一个类做太多事）
- ✅ 开闭原则（修改而非扩展）
- ✅ 依赖倒置（依赖具体而非抽象）
- ✅ 接口隔离（接口过大）
- ✅ 高耦合低内聚

**关键词**：
- SOLID原则
- 单一职责、职责不清
- 高耦合、紧耦合
- 过度设计、设计不足

**典型场景**：
```
✅ 代码质量问题：
  class UserService {
      void createUser() {}
      void sendEmail() {}  // 应该在EmailService
      void generateReport() {}  // 应该在ReportService
  }
  // 违反单一职责

✅ 代码质量问题：
  if (type.equals("A")) {
      // 处理A
  } else if (type.equals("B")) {
      // 处理B
  }
  // 每增加一个类型都要修改，违反开闭原则

✅ 代码质量问题：
  class OrderService {
      private MySQLOrderDao dao = new MySQLOrderDao();
      // 依赖具体实现，违反依赖倒置
  }

❌ 不是设计问题（过度设计）：
  // 简单的CRUD非要搞一堆设计模式
```

**判断标准**：
- **是否影响扩展？** 新增功能需要大改
- **是否职责混乱？** 一个类做多个不相关的事
- **是否耦合过高？** 修改一处影响多处

## 分类5：🎨 代码风格

### 核心判断标准

**关键问题：这个问题仅影响代码可读性/一致性吗？**

如果答案是"是"，且不影响功能/质量，则归类为代码风格。

### 判断依据

| 判断维度 | 是否属于代码风格 |
|---------|----------------|
| **仅是格式问题** | ✅ 是 |
| **不影响功能** | ✅ 是 |
| **不影响可维护性** | ✅ 是 |
| **影响可读性** | ⚠️ 可能是 |
| **影响理解** | ❌ 否（代码质量） |
| **影响功能** | ❌ 否（功能问题） |

### 子类型判断

#### 1格式问题

**核心特征**：
- ✅ 缩进不一致
- ✅ 空格、换行不规范
- ✅ 括号位置不统一
- ✅ 行长度超限

**关键词**：
- 格式、缩进、空格
- 换行、行长度

**典型场景**：
```
✅ 代码风格问题：
  if(a){  // 缺少空格
  }

✅ 代码风格问题：
      if (a) {  // 缩进4空格
    }           // 缩进2空格

❌ 不是风格问题（是质量问题）：
  嵌套过深导致难以阅读 → 复杂度高
```

---

#### 2. 注释问题

**核心特征**：
- ✅ 复杂逻辑缺少注释
- ✅ 注释与代码不一致
- ✅ 过多无意义注释
- ✅ 缺少类/方法文档

**关键词**：
- 注释、文档、JavaDoc

**典型场景**：
```
✅ 代码风格问题：
  // 计算价格
  double result = price * 0.8;  // 实际是打8折
  // 注释与代码不符

✅ 代码风格问题：
  public void complexCalculate(...) {
      // 100行复杂计算逻辑，无注释
  }

❌ 不是注释问题（不重要）：
  简单的getter/setter缺少注释
```

**判断标准**：
- **是否影响理解？** 复杂逻辑需要注释
- **是否必要？** 自解释代码不需要注释

---

#### 3. 命名风格

**核心特征**：
- ✅ 驼峰/下划线混用
- ✅ 常量未全大写
- ✅ 类名不符合规范
- ✅ 包名不规范

**关键词**：
- 命名风格、命名规范
- 驼峰、下划线

**与"命名问题"区别**：
- **命名风格**：违反格式规范，但语义明确
- **命名问题**：语义不明确或不符

**典型场景**：
```
✅ 代码风格问题：
  int userAge;  // 驼峰
  int user_name;  // 下划线
  // 风格不一致

✅ 代码风格问题：
  final String apiKey = "...";
  // 常量应该 API_KEY

❌ 不是风格（是质量问题）：
  int a;  // 无意义命名 → 命名问题
```

---

#### 4. 导入管理

**核心特征**：
- ✅ 导入顺序混乱
- ✅ 未使用的导入
- ✅ 通配符导入（import xxx.*）
- ✅ 导入内部实现类

**关键词**：
- import、导入
- 未使用导入、通配符导入

**典型场景**：
```
✅ 代码风格问题：
  import java.util.*;  // 通配符

✅ 代码风格问题：
  import com.example.Unused;  // 未使用

❌ 不重要（可忽略）：
  导入顺序不符合某种规范（如果不影响可读性）
```

## 边界案例分析

### 案例1：缺少参数校验

**场景**：接口方法未校验参数是否为空

```java
public void updateUser(Long userId, String name) {
    User user = userDao.findById(userId);
    user.setName(name);
    userDao.update(user);
}
```

**分类决策**：
1. 是否涉及安全？
   - 如果是**权限校验**缺失 → 🔒 **安全问题-权限校验**
   - 如果是**业务参数校验**缺失 → 继续判断
2. 是否影响功能？
   - 会导致NullPointerException → 📌 **功能问题-边界条件**

**结论**：📌 功能问题-边界条件

---

### 案例2：SQL字符串拼接

**场景1**：拼接用户输入
```java
String sql = "SELECT * FROM users WHERE name = '" + userName + "'";
// userName 来自用户输入
```
**结论**：🔒 **安全问题-注入攻击**（用户可控）

**场景2**：拼接常量
```java
String sql = "SELECT * FROM users WHERE status = '" + UserStatus.ACTIVE + "'";
// UserStatus.ACTIVE 是常量
```
**结论**：📝 **代码质量-设计原则**（不是安全问题，但代码质量差）

**关键判断**：用户输入是否可控？

---

### 案例3：资源未关闭

**场景1**：导致功能失败
```java
FileOutputStream fos = new FileOutputStream(file);
fos.write(data);
// 未关闭导致文件被锁，其他进程无法访问
```
**结论**：📌 **功能问题-异常处理**（影响功能）

**场景2**：仅影响资源
```java
FileInputStream fis = new FileInputStream(file);
// 读取后未关闭，导致文件句柄泄漏
```
**结论**：⚡ **性能问题-内存泄漏**（仅影响资源）

**关键判断**：是否影响功能正确性？

---

### 案例4：日志打印数据

**场景1**：打印敏感信息
```java
log.info("用户登录：userId={}, password={}", userId, password);
```
**结论**：🔒 **安全问题-敏感信息**

**场景2**：打印普通信息
```java
log.info("处理订单：orderId={}", orderId);
```
**结论**：✅ **不是问题**

**关键判断**：是否是敏感信息？

---

### 案例5：循环查询数据库

**场景1**：循环次数不确定
```java
List<Order> orders = getOrders();  // 可能有成千上万条
for (Order order : orders) {
    User user = userDao.findById(order.getUserId());
}
```
**结论**：⚡ **性能问题-查询优化**（N+1问题）

**场景2**：循环次数固定且少
```java
List<String> codes = Arrays.asList("A", "B", "C");  // 固定3个
for (String code : codes) {
    Dict dict = dictDao.findByCode(code);
}
```
**结论**：📝 **代码质量-设计原则**（可优化但影响不大）

**关键判断**：数据量是否可能很大？

---

### 案例6：变量命名不当

**场景1**：语义不明
```java
List<Object> data = getData();
int a = calculate(b);
```
**结论**：📝 **代码质量-命名问题**

**场景2**：仅是风格不符
```java
int user_age;  // 应该用驼峰
final String apiKey = "...";  // 常量应该大写
```
**结论**：🎨 **代码风格-命名风格**

**关键判断**：是否影响理解？

---

### 案例7：嵌套过深

**场景1**：影响阅读和维护
```java
if (a) {
    if (b) {
        if (c) {
            if (d) {
                // 业务逻辑
            }
        }
    }
}
```
**结论**：📝 **代码质量-复杂度高**

**场景2**：仅是格式问题
```java
if(a){if(b){...}}  // 格式不对但逻辑简单
```
**结论**：🎨 **代码风格-格式问题**

**关键判断**：是否影响可维护性？

## 多类型问题的优先级

当一个问题同时符合多个分类时，按以下优先级选择：

```
🔒 安全问题 > 📌 功能问题 > ⚡ 性能问题 > 📝 代码质量 > 🎨 代码风格
```

### 优先级示例

**示例1**：缺少权限校验
- 可以是"功能遗漏"（缺少功能）
- 可以是"安全问题"（存在安全风险）
- **最终分类**：🔒 **安全问题-权限校验**（优先级高）

**示例2**：空指针异常
- 是"功能问题"（会导致NPE）
- 不涉及安全
- **最终分类**：📌 **功能问题-边界条件**

**示例3**：N+1查询
- 功能正确但性能差
- 不涉及功能和安全
- **最终分类**：⚡ **性能问题-查询优化**

## 常见错误分类

### ❌ 错误1：SQL拼接一律归为安全问题

**错误示例**：
```java
String sql = "SELECT * FROM config WHERE key = '" + CONFIG_KEY + "'";
// CONFIG_KEY是常量，归为"安全问题"
```

**正确分类**：
- 如果拼接常量 → 📝 **代码质量问题**
- 如果拼接用户输入 → 🔒 **安全问题**

**判断依据**：用户是否可控

---

### ❌ 错误2：所有命名问题归为代码风格

**错误示例**：
```java
List<Object> data = getData();  // 归为"代码风格"
```

**正确分类**：
- 如果语义不明 → 📝 **代码质量-命名问题**
- 如果仅是风格 → 🎨 **代码风格-命名风格**

**判断依据**：是否影响理解

---

### ❌ 错误3：所有循环查询归为性能问题

**错误示例**：
```java
for (int i = 0; i < 3; i++) {  // 固定3次
    query();
}
// 归为"性能问题"
```

**正确分类**：
- 如果数量级大 → ⚡ **性能问题**
- 如果数量级小且固定 → 📝 **代码质量**（影响不大）

**判断依据**：数据量是否可能很大

---

### ❌ 错误4：缺少校验都归为功能问题

**错误示例**：
```java
public void deleteUser(Long userId) {
    userDao.delete(userId);  // 缺少权限校验
}
// 归为"功能问题-功能遗漏"
```

**正确分类**：
- 如果是权限校验 → 🔒 **安全问题-权限校验**
- 如果是参数校验 → 📌 **功能问题-边界条件**

**判断依据**：是否涉及安全

## AI分类指导原则

### 原则1：优先看本质影响

不要被表面现象迷惑，看问题的**本质影响**：
- 会导致安全风险 → 安全问题
- 会导致功能错误 → 功能问题
- 仅影响性能 → 性能问题
- 仅影响质量 → 代码质量
- 仅影响格式 → 代码风格

### 原则2：关注影响范围

- **安全**：可能被利用、造成损失
- **功能**：特定场景下必然失败
- **性能**：数据量大时性能下降
- **质量**：长期累积影响维护
- **风格**：几乎无实际影响

### 原则3：考虑用户可控性

很多问题的分类取决于"用户是否可控"：
- SQL拼接：用户可控 → 安全；常量 → 质量
- 参数校验：权限 → 安全；业务 → 功能

### 原则4：保持一致性

**同类问题应有相同分类。** 判断新问题时，参考已分类的相似问题，确保一致性。

## 总结

### 分类关键词速记

| 分类 | 关键特征 | 典型关键词 |
|-----|---------|-----------|
| 🔒 安全 | 可被利用、权限、敏感 | 注入、越权、泄露、弱加密 |
| 📌 功能 | 功能错误、会失败 | 逻辑错误、空指针、异常、遗漏 |
| ⚡ 性能 | 响应慢、资源耗尽 | N+1、泄漏、全表扫描、未缓存 |
| 📝 质量 | 难维护、难扩展 | 重复代码、复杂度、命名、设计 |
| 🎨 风格 | 仅格式、不影响逻辑 | 缩进、注释、命名风格、导入 |

### 分类核心问题

1. **安全吗？** → 🔒 安全问题
2. **功能对吗？** → 📌 功能问题
3. **性能好吗？** → ⚡ 性能问题
4. **好维护吗？** → 📝 代码质量
5. **格式对吗？** → 🎨 代码风格

### 记住优先级

**安全 > 功能 > 性能 > 质量 > 风格**

当问题符合多个分类时，选择优先级最高的。
