# 内网同步方案 - 离线包分发方案

## 方案概述

**方案类型：** Python包离线分发 + 智能同步

**核心思路：** 
1. 内网文件服务器从GitHub拉取最新代码
2. 构建wheel包并下载Python依赖
3. 通过HTTP文件服务器提供下载
4. 内网开发者使用uv工具离线安装

**适用场景：**
- 内网服务器能够访问外网GitHub
- 内网开发者**无法**访问外网PyPI
- 需要完全离线的安装体验

**方案优势：**
- ✅ 完全离线：内网开发者无需访问外网
- ✅ 智能更新：代码和依赖分离，避免重复下载
- ✅ 安装简单：一条命令完成安装
- ✅ 维护便捷：定时任务自动化同步

---

## 架构演进说明

### 初始方案（已废弃）
GitHub → GitLab镜像 → 内网用户从GitLab安装

**废弃原因：** 
- uv工具从Git安装时仍需访问PyPI下载依赖包
- 无法解决内网用户访问外网的问题

### 最终方案（当前）
GitHub → 构建wheel包 + 下载依赖 → HTTP文件服务器 → 完全离线安装

**核心改进：**
- 预先下载所有依赖包到文件服务器
- 内网用户使用 --find-links + --no-index 完全离线安装

## uv工具安装原理分析

### uv tool install 的工作流程

当执行命令：`uv tool install specify-cli --from git+https://...`

**执行步骤：**
1. ✅ 克隆Git仓库（从指定的Git地址）
2. ✅ 读取pyproject.toml配置
3. ❌ **从PyPI下载依赖包**（需要外网访问）
4. ✅ 构建wheel包
5. ✅ 安装到隔离环境

**关键问题：** specify-cli依赖11个Python包：
- typer
- click>=8.1
- rich
- httpx[socks]
- platformdirs
- readchar
- truststore>=0.10.4
- pyyaml>=6.0
- packaging>=23.0
- pathspec>=0.12.0
- json5>=0.13.0

**即使代码在内网，uv仍会尝试从PyPI下载这些依赖包！**

### 离线安装解决方案

**核心参数：**
- `--find-links <URL>` - 指定包查找路径（可多次使用）
- `--no-index` - 禁止访问PyPI（必须）

**完整命令示例：**
```bash
uv tool install specify-cli \
  --find-links http://172.16.37.100:8888/packages/spec-kit \
  --find-links http://172.16.37.100:8888/packages/dependencies \
  --no-index
```

这样uv只会从指定的文件服务器查找包，不会访问外网PyPI。

### 设计原则

**问题：** 依赖包版本稳定，无需每次都重新下载

**解决方案：** 项目代码和依赖包分离存储

### 访问地址映射

| 文件类型 | 服务器路径 | HTTP访问地址 |
|---------|-----------|-------------|
| 项目wheel包 | /opt/spec-kit-packages/packages/spec-kit/ | http://172.16.37.100:9999/packages/spec-kit/ |
| 依赖包 | /opt/spec-kit-packages/packages/dependencies/ | http://172.16.37.100:9999/packages/dependencies/ |
| 安装脚本 | /opt/spec-kit-packages/sync-workspace/ | http://172.16.37.100:9999/sync-workspace/ |

### 分离存储的优势

**场景1：日常代码更新（99%的情况）**
- 检测GitHub是否有新commit
- 如果有：构建wheel包 → 复制到packages/spec-kit/
- dependencies/目录不变
- **时间：** ~2-3分钟

**场景2：依赖版本升级（1%的情况）**
- 检测pyproject.toml的[project.dependencies]变化
- 如果变化：重新下载所有依赖包 → 复制到packages/dependencies/
- **时间：** ~5分钟（额外）

**优势总结：**
- ✅ 节省带宽：依赖包只下载一次
- ✅ 加快同步：大部分时间只需构建项目包
- ✅ 存储优化：避免重复存储相同依赖
- ✅ 灵活更新：项目代码和依赖独立管理

### 步骤3：创建HTTP服务启动脚本

**创建启动脚本：** /opt/spec-kit-server/start-server.sh

```bash
cat > /opt/spec-kit-server/start-server.sh << 'EOF'
#!/bin/bash

# spec-kit HTTP文件服务器启动脚本
# 端口：9999
# 工作目录：/opt/spec-kit-packages

PORT=9999
WORK_DIR="/opt/spec-kit-packages"
LOG_FILE="/opt/spec-kit-server/server.log"
PID_FILE="/opt/spec-kit-server/server.pid"

# 检查是否已经运行
if [ -f "$PID_FILE" ]; then
    PID=$(cat "$PID_FILE")
    if ps -p "$PID" > /dev/null 2>&1; then
        echo "服务已经在运行中 (PID: $PID)"
        exit 1
    fi
fi

# 启动服务
cd "$WORK_DIR"
echo "========================================"
echo "spec-kit HTTP File Server Started"
echo "URL: http://172.16.37.100:$PORT"
echo "Working Directory: $WORK_DIR"
echo "Log File: $LOG_FILE"
echo "========================================"

nohup python3 -m http.server $PORT --bind 0.0.0.0 > "$LOG_FILE" 2>&1 &
echo $! > "$PID_FILE"

echo "服务启动成功！PID: $(cat $PID_FILE)"
echo "访问地址: http://172.16.37.100:$PORT"
EOF

chmod +x /opt/spec-kit-server/start-server.sh
```

**创建停止脚本：** /opt/spec-kit-server/stop-server.sh

```bash
cat > /opt/spec-kit-server/stop-server.sh << 'EOF'
#!/bin/bash

PID_FILE="/opt/spec-kit-server/server.pid"

if [ ! -f "$PID_FILE" ]; then
    echo "服务未运行或PID文件不存在"
    exit 1
fi

PID=$(cat "$PID_FILE")
if ps -p "$PID" > /dev/null 2>&1; then
    kill "$PID"
    rm "$PID_FILE"
    echo "服务已停止 (PID: $PID)"
else
    echo "进程不存在，清理PID文件"
    rm "$PID_FILE"
fi
EOF

chmod +x /opt/spec-kit-server/stop-server.sh
```

**启动服务：**

```bash
# 启动spec-kit服务器（9999端口）
/opt/spec-kit-server/start-server.sh

# 验证服务运行
```

## 风险与限制

### 存在的限制

**1. 同步延迟**
- 定时任务有时间间隔（建议1小时）
- 紧急更新需要手动触发

**应对措施：**
- 提供手动触发脚本
- 重要版本提前通知

**2. 网络依赖**
- 依赖文件服务器访问GitHub
- 外网不稳定时可能同步失败

**应对措施：**
- 配置重试机制
- 保留上一个可用版本
- 设置失败告警

**3. 存储空间**
- wheel包持续accumulate会占用空间
- 建议定期清理旧版本

**应对措施：**
- 只保留最近N个版本
- 配置自动清理脚本

**4. 单点故障**
- 文件服务器故障影响所有用户

**应对措施：**
- 配置服务器监控
- 提供备用下载方式（如共享目录）

### 风险评估矩阵

| 风险项 | 概率 | 影响 | 优先级 | 缓解措施 |
|--------|------|------|--------|----------|
| 同步延迟 | 高 | 低 | 中 | 手动触发 + 提前通知 |
| 网络故障 | 中 | 中 | 中 | 重试机制 + 告警 |
| 存储满 | 低 | 中 | 低 | 自动清理旧版本 |
| 服务器宕机 | 低 | 高 | 高 | 监控 + 备用方案 |

## 故障排查指南

### 问题1：同步脚本执行失败

**症状：** cron任务执行但无输出

**排查步骤：**
```bash
# 1. 检查脚本权限
ls -l /opt/ai-cr/sync-workspace/sync-spec-kit.sh

# 2. 手动执行查看错误
bash -x /opt/ai-cr/sync-workspace/sync-spec-kit.sh

# 3. 检查日志
tail -f /opt/ai-cr/logs/sync.log
```

**常见原因：**
- 脚本没有执行权限（chmod +x）
- PATH环境变量不正确
- uv工具未正确安装

---

### 问题2：内网用户安装失败

**症状：** uv tool install 报错

**排查步骤：**
```bash
# 1. 验证文件服务器访问
curl http://172.16.37.100:8888/packages/spec-kit/

# 2. 检查依赖包是否完整
curl http://172.16.37.100:8888/packages/dependencies/

# 3. 尝试详细输出模式
uv tool install specify-cli -v \
  --find-links http://172.16.37.100:8888/packages/spec-kit \
  --find-links http://172.16.37.100:8888/packages/dependencies \
  --no-index
```

**常见原因：**
- 缺少 `--no-index` 参数，仍尝试访问PyPI
- 依赖包未完整下载
- uv版本过旧

---

### 问题3：包版本不匹配

**症状：** 安装提示依赖版本冲突

**解决方案：**
```bash
# 在服务器上重新下载完整依赖
cd /opt/ai-cr/sync-workspace/spec-kit-repo
rm -rf /opt/ai-cr/packages/dependencies/*
uv pip download . -d /opt/ai-cr/packages/dependencies/
```
